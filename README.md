# 議会一般質問ダッシュボード

> 議会だよりに載る「一般質問」を、**議員別・テーマ別に横断して見渡せる**ようにする、オープンソースの静的サイトとオープンデータです。
> まずは日野町議会を対象に作成し、他の自治体議会でもそのまま使えるよう設計しています。

> **位置づけ（重要）**　本リポジトリは現時点では一個人による**提案・試作**です。議会・広報委員会の公式な成果物として採用されたものではありません。将来、広報委員会の取り組みとして位置づけられることを目指していますが、公開・運用・選挙期間の利用にあたっては議会事務局・選挙管理委員会の確認を前提とします。

---

## なぜ作るのか

議会だよりは毎号きちんと発行され、一般質問も掲載されています。けれども紙やPDFの号をまたいで「この議員は任期を通じて何に取り組んできたか」「町ではどのテーマが議論されているか」を**横断して見渡す**手段がありません。号ごとにPDFを開いて読み比べるしかない状態です。

このダッシュボードは、すでに公開されている議会だより・会議録を、見方を変えて整理し直すだけのものです。新しい情報を足すのではなく、**既にある一次情報へたどり着きやすくする**ことを目的にしています。

## できること

- **議員別**：一人ひとりが、いつ・何を質問し、町がどう答えたかを一覧。原文と、会議録・録画へのリンク付き。
- **テーマ別**：子育て・教育、農林業、防災など、どの分野が議論されているかを俯瞰。
- **質問一覧**：全質問を検索・絞り込み（号／議員／テーマ／掲載・未掲載）。
- **原文表示**：議会だより掲載のままの問・答を再現（要約ではなく原文）。
- **一次情報への動線**：各質問から議会だより（PDF）・会議録・録画配信へ。

## 質問数や並べ替えについて

号・議員・テーマでの絞り込みや並べ替えは、技術的に行えるようにしてあります。件数での並べ替えも仕組みとしては可能です。一方で、件数をそのまま議員間で比べると、読み手に一面的な印象を与えてしまうことがあります。たとえば在任期間が異なる場合（補欠当選などで登壇できた回数が違う）、登壇しない期間がある場合、1つの論点を①②③と分ける議員と1問にまとめる議員とで数え方が揃わない場合などです。

そこで既定の並びは議席番号順にし、件数を大きく順位付けする見せ方は初期表示にしていません。件数で並べ替えたときは、その旨と上の注意が併記されるようにしています。件数を隠すためではなく、探索の自由は保ちつつ、初見の印象が偏らないようにするための「見え方」への配慮です。

こうしたダッシュボードを実装するときに考えた見え方の注意点やテーマ分類の考え方は [docs/EDITORIAL_POLICY.md](docs/EDITORIAL_POLICY.md)、設計上の検討と修正計画は [docs/DESIGN_REVIEW.md](docs/DESIGN_REVIEW.md) にまとめています。

## データの出典と限界

データの出典は各議会の**議会だより**と**会議録**です。ただし議会だより掲載文は紙面の都合で**編集・要約された版**であり、全文ではありません。やりとりの全文は会議録・録画にあたれるよう、すべての質問にリンクを用意しています。

また、3問以上した議員の質問は、議会だよりでは〇の付いた番号だけが掲載されます（紙面の都合）。本データでは未掲載の質問も題名だけは収録し、「掲載」と明確に区別したうえで、本文は会議録で確認できるよう案内します。

## 使い方

**見る人**：公開サイト（GitHub Pages）をブラウザで開くだけです。インストールは不要です。

**新しい号を追加する人**：`data/_template.json` をコピーして `data/issue-NN.json` を作り、議会だより・会議録から問・答を流し込みます。既存の号には触りません。手順は [docs/ADD_NEW_ISSUE.md](docs/ADD_NEW_ISSUE.md)、データの形式は [docs/SCHEMA.md](docs/SCHEMA.md) を参照。

**他の自治体で使う人**：`config.json`（自治体名・議員・テーマ・会期）と `data/`（各号）を自分の議会のものに差し替えると動きます。表示される自治体名・会期・出典などはすべて `config` と `data` から生成され、テンプレート（`src/render_html.py`）に自治体固有の文言は埋め込んでいません。

## 似た取り組みとの関係

「議会だよりの一般質問を、議員別・テーマ別に横断して見せる」ものに完全一致する事例は、調査の範囲（2026-06、見落としの可能性あり）では見当たりませんでした。隣接する取り組みは次のとおりです。

- **公式の会議録検索システム（ASP型）** — 本会議・委員会の全文をキーワード検索・閲覧。多くの市町村議会が導入（滋賀県内では[愛荘町](https://www.town.aisho.shiga.jp/gikai/kaigiroku/index.html)など）。
  - [会議録研究所「Discuss」](https://www.kaigiroku.co.jp/contents/gikai/) ／ [大和速記「DB-Search」](https://www.yamatosokki.co.jp/service/db_search/) ／ [kaigiroku.net（NTT-AT系）](https://ssp.kaigiroku.net/)
- **シビックテックの横断検索** — 複数自治体をまたいだ議事録の全文検索。[chiholog](https://chiholog.net/chiholog)
- **研究プロジェクト** — 会議録の言語分析。47都道府県・東京23区の「議員の特徴語」(TF-IDF)等を公開。[地方議会会議録コーパスプロジェクト（local-politics.jp）](http://local-politics.jp/)
- **国レベルのオープンデータ＋静的サイト** — 国会議案・賛否を CSV/JSON で公開し、GitHub Pages で閲覧、GitHub Actions で自動更新。[スマートニュース「国会議案データベース」](https://smartnews-smri.com/research/research-1127/)（[GitHub](https://github.com/smartnews-smri/house-of-representatives) ／ [閲覧ページ](https://smartnews-smri.github.io/house-of-representatives/)）
- **議会DX・広報の改善事例** — 議会広報の配信・伝達手段の改善。[総務省「地方議会デジタル化の事例」](https://www.soumu.go.jp/main_content/000881321.pdf)（神奈川県開成町の一般質問予告/感想動画ほか）

なお、会議録のような一次データが機械判読可能な形で公開されていれば、この種のダッシュボードはもっと簡単に作れます。実際には編集済みの議会だより（PDF）から人手で起こしており、全文を持つ会議録検索システムも、機械判読できるオープンデータとしては開かれていないことが多いのが実情です。本企画は当面、その不足を手作業で補う位置づけです。

公開のかたち（機械判読データ＋静的サイト＋自動更新）は、官公庁が分散公開していた情報を第三者が整理して二次利用可能にした「[国会議案データベース](https://smartnews-smri.com/research/research-1127/)」（スマートニュース）を参考にしています。

## 設計で参考にした指針

画面の設計にあたっては、デジタル庁「[ダッシュボードデザインの実践ガイドブック](https://www.digital.go.jp/resources/dashboard-guidebook)」を参考にしています。色やテンプレート（Power BI）への準拠ではなく、**要件定義（誰が・なぜ・何を見るか）、情報の選別、過度な装飾を避けること、提示型と探索型の区別**といった設計プロセスと情報整理の考え方を取り入れています。ガイドブックを物差しにしたUI/UX・情報整理・情報管理のレビューと、その修正計画は [docs/DESIGN_REVIEW.md](docs/DESIGN_REVIEW.md) にまとめています。

## リポジトリ構成

```
config.json              自治体プロファイル（議員・テーマ・会期・方針）
data/issue-NN.json       議会だより1号ぶん（問・答の本文込み）
data/_template.json      新しい号を作るときのひな形
build.py                 config + data から docs/ を生成（ダッシュボード＋オープンデータ）
src/render_html.py       ダッシュボードHTMLのレンダラ（build.py が呼び出す）
docs/                    GitHub Pages の配信対象（index.html ＋ data/ のCSV・JSON）
docs/data/               オープンデータ（questions.csv/json, members.csv, themes.csv, issues.csv）
docs/EDITORIAL_POLICY.md 編集方針（質問数の扱い・テーマ分類・出典など）
docs/DESIGN_REVIEW.md    設計レビューと修正計画（デジタル庁ガイドブック準拠の観点）
docs/SCHEMA.md           データ辞書
docs/ADD_NEW_ISSUE.md    新号の追加手順
scripts/validate.py      データ検査（キー重複・テーマ名・掲載なのに本文空 などを点検）
.github/workflows/       push で 検査→ビルド→Pages公開 を自動実行
```

ビルドは `python build.py`（事前に `python scripts/validate.py` で検査）。GitHub Pages は、リポジトリの Settings → Pages で配信元を「GitHub Actions」に設定すると、`config.json`・`data/` の変更を push するだけで自動更新されます。

## オープンデータ

`docs/data/` に、二次利用しやすいCSV・JSON（質問・議員・テーマ・号の各テーブルと集計）を出力します。データ辞書とライセンスを同梱します。
