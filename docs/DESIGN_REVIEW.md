# 設計レビューと修正計画

デジタル庁「[ダッシュボードデザインの実践ガイドブック](https://www.digital.go.jp/resources/dashboard-guidebook)」を物差しに、UI/UX・情報整理・情報管理の観点で点検した記録と、改善計画。色やテンプレート（Power BI）への準拠ではなく、**設計プロセスと情報の扱い方**の指針として参照している。

## 前提：このダッシュボードの性質

ガイドブックが主に想定するのは、数値・指標を監視する「提示型（モニタリング型）」ダッシュボードである。本企画は数値の監視ではなく、**質的なテキスト（問・答）を議員別・テーマ別にたどる記録ナビゲーション**が主で、数値は副次的。したがってガイドの原則は、当てはまるもの（要件定義・情報の選別・粒度・鮮度・装飾の抑制）を採り、KPIチャート中心の部分はそのままは採らない。

ガイドブックは提示型と探索型を区別する。本企画では概要＝提示型（全体をざっと把握）、質問一覧・議員×テーマ＝探索型（絞り込み・掘り下げ）と整理できる。この役割分担を明示する。

## 観点別レビューと修正計画

### 1. 目的・対象（5W1H）
- 現状: 冒頭に「誰のための・何を見るものか」（有権者／議員／役場）を明記済み。ガイドの要件定義の考え方を概ね満たす。
- 修正: 提示型（概要）と探索型（一覧・マトリクス）の役割を、見出しか短い導入文で明示する。〔小〕

### 2. 「違いに気づける」原則と中立性の調整
- ガイドは「目立つデータ・異常値に気づけるように」とするが、本企画では件数で議員を序列化しない方針のため、これを**件数の大小ではなくテーマの広がり等の質的側面**に読み替える。
- 修正: 概要のテーマ表示で「多い順・％」を強調しない方向に再構成（テーマ再編の検討と連動）。件数は事実として残すが主役にしない。〔中〕

### 3. 載せる情報の選別（ノイズの削除）
- 現状: 影・グラデーション等の装飾がやや多い。テキスト主体の画面では可読性を下げうる。
- 修正: 装飾の棚卸し。意味のない影・グラデーションを削減し、色は「テーマの識別」など意味のある箇所に限定する（色を消す/青にする話ではなく、装飾過多の整理）。〔中〕

### 4. グラフ選択
- 現状: 「定例会ごとの質問数」を棒で表現。議会全体の活動量の分布であり、議員間比較ではないので問題は小さい。ガイドは時間変化は折れ線を推奨。
- 修正: 件数を強調しない方針に合わせ、この時系列の意図（活動の分布／各号への入口）を明確化。必要なら折れ線も検討。〔小〕

### 5. 分解できる（粒度・ドリルダウン）
- 現状: テーマ→質問、議員→質問、マトリクスのセル→絞り込み、と多段で掘り下げ可能。ガイドの原則を満たす。維持。

### 6. 鮮度（定期更新）
- 現状: 号別JSON＋ビルドで更新可能。
- 修正: 画面に「対象期間・最終更新日」を明示。更新手順は `docs/ADD_NEW_ISSUE.md` に整備済み。〔小〕

### 7. アクセシビリティ（参考: デジタル庁「ウェブアクセシビリティ導入ガイドブック」）
- 現状: タブの `aria-selected`、画像の `alt`、テーマは色＋ラベル併記（色のみ依存を回避）など一定の配慮あり。
- 修正: コントラスト比の点検、キーボード操作とフォーカス可視化、モーダルの読み上げ順、`title` 依存を `aria-label` 等で補強。〔中〕

### 8. 情報管理（データガバナンス）
- 現状: 出典・未掲載の区別・スキーマ・検査スクリプトを明文化済み。ガイドが「データ準備は範囲外」とする部分を、本企画はむしろ強みとして整備している。維持。

## 修正の優先順位（たたき台）
1. 〔中〕装飾の棚卸し（ノイズ削除）
2. 〔中〕概要のテーマ表示の再構成（件数強調の緩和・テーマ再編と連動）
3. 〔中〕アクセシビリティ点検
4. 〔小〕提示型/探索型の役割明示、時系列の意図明確化、最終更新日の表示

## 国会議案データベース（スマートニュース）からの援用と、逆に提案しうること

「[国会議案データベース](https://smartnews-smri.com/research/research-1127/)」は、官公庁が回次ごと・議案ごとに分散公開していた情報を、第三者が機械判読可能な CSV/JSON に集約し、MIT で無償公開、GitHub Pages で閲覧、GitHub Actions で自動更新する取り組み。報道・研究の二次利用を可能にすることが狙い。

- 援用できる点: 「機械判読データ＋静的サイト＋自動更新＋明確なライセンス」という公開の作法。閲覧UIとデータの分離、出典明記、スキーマ設計。本企画の `docs/data`（CSV/JSON）と Actions 構成はこの作法に倣う。
- 逆に提案しうる点: 地方版の同種データが各地に増えた場合、議員ID・会期ID・テーマ語彙などの規格を揃えれば、横断検索（chiholog 等）や会議録コーパスと相互運用できる。地方議会の「一般質問」を構造化した粒度は既存の議案DBにはないため、地方版オープンデータの共通スキーマづくりに寄与しうる。

## 実装状況（このリポジトリ）

- 〔済〕`build.py` 化：config + 号別JSON から `docs/index.html` とオープンデータ（CSV/JSON）を生成。`scripts/validate.py` で検査。GitHub Actions で push 時に自動ビルド・公開。
- 〔済〕装飾の棚卸し：ヘッダ・タイムライン・モーダルのグラデーションと斜線ストライプを平坦化（議長＝0問のハッチなど、意味のある表現は維持）。
- 〔済〕概要のテーマ表示：「多い順」表記と割合（％）の強調を撤去。件数は事実として残す。
- 〔済〕件数での並べ替え時に、優劣指標ではない旨の注意書きを自動表示。
- 〔済〕提示型／探索型の役割をビュー冒頭に明示。最終更新日をフッターに表示。
- 〔済〕アクセシビリティ：本文コントラストの調整、フォーカスリングの明示、動的に生成される操作要素のキーボード操作（Tab移動・Enter/Spaceで実行）。
- 〔残〕テーマ分類の再編（政策領域でのグルーピング、議員ごとの扱った分野数）。議員・広報委員会への確認後に反映。
- 〔残〕表示テンプレート（`src/render_html.py`）に残る自治体固有の文言を `config` から差し込む形へ外出し。
- 〔残〕アクセシビリティのさらなる点検（スクリーンリーダーでの読み上げ順、モーダルのフォーカストラップ等）。

## 追加実装（2巡目）

- 〔済〕固有文言の config 外出し：自治体名・会期・議員定数・出典・対象期間などをすべて config／データから生成。テンプレートに自治体固有の文言は残っていない。
- 〔済〕アクセシビリティ追加：モーダルを `role="dialog"` 化し、開いたらフォーカス移動・Tabの閉じ込め（フォーカストラップ）・閉じたらフォーカス復帰。タブとパネルを `aria-controls`／`role="tabpanel"` で紐付け。
- 〔済〕色の点検：テーマのピル文字を、各テーマ色を維持しつつコントラスト 4.5:1 以上に調整（11色すべて適合）。ドット・背景の色は据え置き。近接色のペアは色＋ラベル併記で色のみ依存を回避。
- 〔済〕議員モーダルに「テーマ内訳」（その議員の質問が各分野へどの割合で向けられたかをバー＋件数＋％で表示）。要約指標も中立な「扱った分野数」に変更。
- 〔済〕原文の欠落レビュー：掲載271問を自動走査（文末の途切れ・括弧不整合・極端な短さ・記号ノイズ）。答弁側は問題なし。問側で1件（#29 錦戸Q2）の語順・引用符の乱れを画像で確認し修正。残る括弧フラグは半角｢｣と全角」の混在＝議会だより原文のまま。

## メモ：元データ（PDF）の形式について

`#24〜#32` の web 用ファイルは現在、テキストPDFではなく **JPEG画像のZIP**（各ページに画像と簡易テキストを同梱）になっている。`#33〜#35` は通常のテキストPDF。再取り込み・再検証の際は、画像PDFからのテキスト化が必要になる点に留意（本ダッシュボードのデータは既に人手で原文化済み）。
